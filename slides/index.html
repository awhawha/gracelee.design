Case 1
Designing a Safer AI Modeling Workflow for Business Analysts
Redesigning the end-to-end model configuration workflow on an enterprise AutoML platform, so business analysts could build prediction models without needing data science expertise.
Role
Senior product designer
Team
2 designers · 7 engineers
Focus
Workflow redesign · Complex systems UX
The problem
Business analysts were flying blind
Each step looked independent, but changes in one silently affected others downstream. Models trained on the wrong data. Users kept backtracking, unsure what they'd broken or where to fix it.
Who
Business analysts building prediction models, with no data science background to rely on
What was broken
Each step looked independent, but changes in one silently affected others downstream
The loop
Configure params → Run model → Error → Backtrack → Error again
The core insight
Original question
How do we simplify the wizard?
→
Reframed
How do we make the training table visible?
Discovery
BAs were really defining a single flat training table, but the system split this into 5 wizard steps with no visible connection between them. Working sessions with engineers and data scientists helped us map how the ML pipeline actually generates training data — and that reframed the design challenge entirely.
The solution
From wizard to canvas workspace
👁
Dependencies visible upfront
Users can see how each setting affects the training dataset as they configure it
✏️
Inline PTT config
No context switching. Edit Prediction Target Time directly within the workflow
🔗
Auto-connect tables
System scans schema and suggests join keys. Reduces errors for users unfamiliar with database relationships

Key design decision
Why we chose workspace over wizard
Wizard
+ Simpler navigation
+ Each table configured independently
− Dependencies hidden until error
− Forced linear path
− Iteration requires restarting
Workspace Chosen
+ Parameter changes show impact immediately, without restarting
+ System surfaces feedback in context, not at the end
+ Less guesswork about whether the configuration is correct
− Higher initial complexity
− More engineering effort
Validated — usability testing showed users switching panels repeatedly to reconcile parameters, confirming workspace was the right call. Early feedback outweighed layout complexity.
Design detail 01
Inline PTT configuration
PTT appears in the Target panel but is needed later during table connection. Old flow forced users to backtrack across steps to define it. Solution: a click-to-edit panel appears inline. Users define PTT without leaving their current context.
PTT detected → Edit icon appears → Inline panel opens → User defines PTT → Returns to workflow

Design detail 02
Auto-connect tables
BAs were unfamiliar with database join keys. Old flow required manually selecting join columns for each table, causing frequent errors. Solution: system scans schema on table add and suggests the most likely relationship. Users can accept or manually edit.
Table added → Schema scanned → Join suggested → User accepts or edits

Impact
50%reduction in configuration time
10 min → 5 min average
Iteration without restarting
Users could adjust any parameter without starting the workflow over. The CS team flagged this as one of the most noticeable improvements after launch.
The bigger change
Users could adjust any part of the configuration without starting over — the shift from a fragile one-shot process to an iterative one.
What I learned
I came in thinking the fix was to show people more. I left knowing the real question is: does the user actually need to know this, or will it just make them feel like they're doing something wrong?
Case 2
Building a Scalable Design System with Logic-Driven Tokens
Designing a 3-layer token architecture that aligns design and engineering while enabling scalable product development. Built solo.
Role
Solo product designer
Team
2 designers · 7 engineers
Focus
Figma Variables · Dev Mode · Progressive adoption
The problem
The first design system solved fragmentation. But it left behind a new problem.
We had a shared library. But ambiguity and engineering drift were getting worse.
Fragmentation — solved
We had a shared library. Components had a source of truth. Designers weren't recreating things from scratch anymore.
Ambiguity — still there
Two designers, same component, same context, but different tokens. Neither was exactly wrong. The system just didn't encode the right answer.
Engineering drift — getting worse
Engineers started making decisions on their own because asking designers gave inconsistent answers. The gap between design files and production code kept growing.
The architecture
3-layer token architecture
01
Primitive
color-green-700
Raw values. The actual color, spacing, or radius. Nobody picks these directly.
02
Semantic
color.background.brand
Intent-based tokens. "background-brand" tells you what it's for, not what it looks like. Still abstract enough to be misread.
Private
03
Component
--comp-button-bg
The third layer I introduced. "button-bg" tells you exactly where to use it. No guessing.
Key design decision
Why component tokens stay private
Publish globally
+ Every token visible to every designer
− More tokens visible means more decisions to make
− Gets harder to govern as the system scales
− Designers can reach for tokens that were never meant for their context
Keep private (scoped to module) Chosen
+ Narrowed decision space — pick by function, not by guessing which color fits
+ System stays manageable as it grows
+ Token sprawl stays manageable — only repeating patterns earn a component token
− Less visible, requires documentation
Progressive adoption strategy
Adopt gradually. Nothing breaks.
Instead of forcing a big-bang migration, I designed the system to layer onto what engineering already had. The foundation was ready in Figma before engineering had capacity to implement it.
01
Component tokens in Figma
Fully structured and aligned with engineering naming conventions. Designers apply functional logic, not color choices.
02
Inspect via Dev Mode
Engineers inspect token logic directly in Figma Dev Mode when they're ready. No separate spec docs. No waiting for designers to write them.
03
Gradual codebase adoption
When engineering has capacity, tokens replace hard-coded values progressively. Nothing breaks. The foundation is already there.
Governance + engineering handoff
From Figma to code, without the guesswork
Engineers inspect components in Dev Mode and see exact token values. No manual specs, no translation errors.
Result
Faster, more accurate handoff. Engineering questions to design dropped from roughly once a week to once a month.

Impact
Architectural scalability, built in from day one
Streamlined design decisions
Designers stopped second-guessing token choices. Design reviews shifted from token debates to actual design decisions.
Reduced handoff friction
Handoff went from 5 steps to 2. Engineering questions to design dropped from roughly once a week to once a month.
Future-proof architecture
Token architecture supports dark mode and multi-theme without restructuring the foundation. That work is already done.
Reflection
The system ended up being mostly my responsibility to maintain. Without formal contribution guidelines and a clear ownership structure, it was hard to create shared ownership. If I could do it again, I'd establish contribution norms earlier, even informally.
